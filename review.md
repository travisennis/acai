## Findings

- High: resumed and forked sessions currently keep the old `Init` record and append a new one, so the saved JSONL no longer has the required shape of one `Init` at the top followed by conversation records and a terminal `Result`. The old records are seeded straight into the agent stream in [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L245-L248), [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L288-L291), and [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L336-L338). A fresh init is then always emitted in [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L463-L488) via [src/clients/agent.rs](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L133-L140) and [src/clients/agent.rs](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L226-L239). Finally, [src/config/session.rs](file:///Users/travisennis/Projects/cake/src/config/session.rs#L331-L352) writes any existing leading `Init` back unchanged instead of normalizing the file. On `--fork`, this is worse than just duplication: the first line can still contain the parent session ID even though the new file path uses the fork ID, because the new `Session` is created with a fresh ID in [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L339-L342) but that ID is never used to rewrite the first record.

- Medium: the model-lock behavior does not match the plan. The code resolves the current runtime model before loading the session and then rejects any session whose stored model differs from that resolved value in [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L234-L250), [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L277-L293), and [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L325-L342). That means `cake --continue` or `cake --resume ...` will fail if the user's default model changed, even when they did not pass `--model`. The plan explicitly required the opposite behavior: if no `--model` is provided and the session has a stored model, resume with that stored model. The current flow also always constructs the live `Agent` from the pre-resolved config in [src/main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L224-L225), so there is no path that actually switches the runtime model to the session model before the agent starts.
